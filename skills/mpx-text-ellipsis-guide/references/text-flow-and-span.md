# text flow 与 span

## 适用范围

用于判断 `span`、`rich-text`、`special-text` 等 `text-like` 节点在文本流中的职责，尤其是整体文本流省略场景。

## 平台关注点

- 微信小程序 webview 与 支付宝小程序 ali：更接近 CSS 文本流语义。
- 微信小程序 skyline：`span` 可内嵌 `image`，适合表达混排文本流。
- RN：最终需落到 `text` 语义容器，避免按 Web 块级布局心智处理。

## text-like 节点

1. `span` 节点在 skyline 下可内嵌 `image`。它更适合作为文本流容器：在整体文本流省略场景中，可由最外层 `span` 统一承担省略；若目标是块级布局、剩余空间分配或局部片段独立省略，则不应将其视为通用布局容器。
2. `special-text` 和 `rich-text` 为富文本节点在小程序中使用 `span` 包裹，在 RN 中使用 `text` 包裹。其文本内容通过 `text` 属性传入；在省略能力、行数控制等文本行为上，可按 `text` 节点理解。

## 整体文本流省略

- 当目标是“整条混排内容作为一个整体文本流省略”时，可优先让最外层 `span` 统一承担 `max-lines@wx`、`overflow@wx="ellipsis"`，内部 `text`、`image`、`text-like` 节点保持内容语义，无需逐个设置省略。
- 若同时适配 RN，应保证该最外层节点在 RN 下映射为 `text` 语义容器，使整条内容继续作为一个文本流处理。
- 该方案适用于整体省略，不适用于中间片段分别独立省略或外层还要承担复杂块级布局职责的场景。

## 判断信号

- 最外层是否为单个文本流容器，而不是多个独立收缩项。
- 省略目标是否为“整条内容统一省略”，而不是某一段单独省略。
- 是否存在 `image`、`text`、`span`、`rich-text`、`special-text` 等混排片段。

## 处理原则

- 若目标是整体文本流省略，优先让最外层 `span` 或等效文本容器统一承担行数与省略属性，内部节点不单独设置省略。
- 若目标是剩余空间分配、块级边界控制或局部片段独立省略，则让承担布局职责的那一层回到真实 `view`。

## 组件映射

节点添加 `mpxTagName` 属性，用于指定不同平台的映射节点。如 `mpxTagName@wx="span"` 表示小程序侧映射为 `span`，`mpxTagName@ios|android|harmony="text"` 表示 RN 侧映射为 `text`。该写法常用于整体文本流的 `text-like` 混编场景。
