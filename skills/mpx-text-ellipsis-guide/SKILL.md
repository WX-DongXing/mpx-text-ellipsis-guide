# 跨平台文本溢出打点指引

## 目标

1. 对既有布局中文本溢出打点问题进行检测，若存在问题则给出修复建议的代码，遵循最小化改动原则。
2. 根据用户描述的布局给出可完美适配跨端的文本溢出打点解决方案。

## 文本溢出打点原理说明
以下四点均满足时，文本溢出打点才能正常工作：

### 1. 明确边界
文本所在盒子有理论上明确的宽度。

#### 文本处于盒子中的方式
1. 将文本本身变为一个块级元素，跨平台文本块级元素差异见：[文本块级元素差异](references/platform-overflow-differences.md#文本块级元素差异)。
2. 被置于一个块级元素中。

#### 文本所在盒子具有明确宽度
该盒子宽度可以来自固定宽度、父级宽度、剩余空间分配，但必须真实可计算。

### 2. 文本不可无限撑开盒子
文本所在的那个盒子，必须允许自己的宽度小于文本内容本身需要的宽度。

通常有以下几种行为：
1. 文本所处盒子宽度固定，文本随着长度增长直到盒子宽度，再增长则超出盒子宽度。
2. 文本所处盒子宽度可变，盒子随文本长度变化直到盒子明确边界，文本再增长则超出盒子宽度。通常这种盒子会被设置一个明确的 `max-width` 属性，或者处于 flex 布局中并对其设置 `flex-shrink: 1`。

### 3. 指定文本行数
设置预期的文本行数（单行、多行），若未指定则会发生换行，导致文本无法超出盒子宽度。跨平台文本行数差异见：[文本设置行数差异](references/platform-overflow-differences.md#文本设置行数差异)。

### 4. 溢出裁剪
对文本溢出时所处的盒子设置裁剪和打点，跨平台文本溢出裁剪差异见：[文本溢出裁剪差异](references/platform-overflow-differences.md#文本溢出裁剪差异)。

## 行为约束

- 遵循最小化改动原则。
- 不进行与溢出无关的样式调整。
- 不进行较大布局的调整。
- 无法精确检测或存疑时，明确疑问点给用户提示，并根据用户补充的信息再次检测。

## 工作流

1. 根据用户描述判断是修正当前溢出问题还是创建新布局。
2. 如果是修正，则先根据原理进行检测，并对不符合原理的部分进行修复，遵循最小化改动原则。
3. 如果用户未明确指定，默认对第一个 `text` 或者 `text-like` 节点进行修改和适配，默认适配全平台、单行文本溢出打点。
4. 如果省略链路中的某一层使用了 `mpxTagName@wx="span"` 或者 `span` 且内部嵌套了 `special-text`、`rich-text` 节点，同时又承担等建立省略边界的职责，应优先检查该层是否没有真正形成可收缩的布局盒子，并评估是否需要回退为真实 `view`。平台差异见：[text-like 节点的布局边界](references/platform-overflow-differences.md#text-like-节点的布局边界)。
5. 如果是新布局，则根据用户描述的布局给出可完美适配跨端的文本溢出打点解决方案，默认使用 `text` 节点，如果用户指定了 `text-like` 节点，则使用 `text-like` 节点进行适配，最佳实践参考：[常见布局最佳实践](references/layout-overflow-best-practices.md)。

## 输出

以用户指示为准。在输出修改后的代码时，应输出完整的组件代码，避免使用省略号，确保用户可以直接复制或应用修改。在创建新布局时，优先给出指定平台的最佳实践。
